# 历程与促销活动：选择正确的方法 {#journeys-vs-campaigns}

Adobe Journey Optimizer提供了三种强大的方法来吸引和吸引客户。 了解何时使用每种产品是构建有效营销体验的关键。

根据您的特定营销需求，本指南可帮助您在&#x200B;**历程**、**促销活动** （操作和API触发）和&#x200B;**编排的促销活动**&#x200B;之间进行选择。

## 快速比较概述 {#quick-overview}

| 方针 | 最适合 | 执行样式 |
|----------|----------|-----------------|
| **历程** | 带条件逻辑的多步实时客户体验 | 1:1编排 — 每个用户档案按自己的步调 |
| **营销活动（操作和API）** | 简单、计划或触发的消息传递 | 批量或API触发 — 同时所有配置文件 |
| **协调的营销活动** | 具有多实体分段的复杂批处理工作流 | 批次画布 — 所有配置文件一起处理 |

## 详细比较 {#detailed-comparison}

使用本综合表了解主要区别：

| 功能 | 历程 | 营销活动（操作和API触发） | 精心策划的营销活动 |
|---------|----------|-----------------------------------|----------------------|
| **主要目的** | 使用实时客户上下文进行多步骤1:1编排 | 向受众投放一次性或定期消息 | 具有复杂分段工作流程的多步骤批量营销活动 |
| **画布类型** | 1:1画布 — 每个配置文件按各自的进度运行 | 无画布 — 单个操作执行 | 批次画布 — 所有配置文件一起处理 |
| **执行流** | 顺序操作，配置文件维护整个历程的状态 | 同时执行到整个受众 | 包含活动和过渡的多步批处理工作流 |
| **进入机制** | 事件、受众、资格、业务事件 | 手动激活、计划或API触发器 | 批处理工作流的计划执行 |
| **数据模型** | 实时个人资料+事件数据 | 配置文件数据+ API有效负荷 | 多实体关系数据（配置文件、产品、商店、预订） |
| **区段** | 预建受众+实时条件 | Experience Platform中的预建受众 | 在画布中构建的按需受众具有确切计数 |
| **配置文件正在处理** | 单个实时（当事件发生时） | 批次，一次完成 | 批处理，以及多实体支持 |
| **个性化** | 实时上下文数据+配置文件属性 | 配置文件属性+ API有效负载数据 | 用于精确定位的多实体数据 |
| **复杂性** | 具有分支、等待时间和条件的多步 | 单个操作或简单工作流 | 具有分段、扩充、拆分的多步骤批处理工作流 |
| **最适合** | 客户生命周期历程、入门、购物车放弃 | 促销活动、新闻稿、公告、事务型消息 | 复杂的季节性促销活动、多步促销活动、产品发布 |
| **计时** | 连续，发布后始终有效 | 计划的开始/结束日期或API触发的日期 | 按计划批量执行 |
| **状态管理** | 维护实时操作的客户状态 | 无状态执行 | 使用工作表进行批处理 |
| **使用时间** | 需要具有实时决策逻辑的多个接触点 | 在特定时间向受众发送简单消息 | 需要复杂的分段、多实体数据或精确的预发送计数 |
| **独特的功能** | 实时反应、等待活动、基于用户档案的步调 | 简单计划、API触发、速率控制 | 关系数据集，多实体分段，精确计数，多级发送 |

## 决策指南 {#decision-guide}

按照此诊断树选择正确的方法：

### 步骤1：您的执行要求是什么？

**个人对客户行为的实时响应？**
→ **使用历程**
- 用户档案需要按照自己的速度移动
- 基于行为的条件逻辑
- 实时上下文至关重要

**是否向受众进行简单消息投放？**
→ **使用操作或API触发的营销活动**
- 所有用户档案都会同时接收消息
- 通过API计划或触发
- 无需复杂的多步逻辑

**具有高级分段的复杂批处理工作流？**
→ **使用协调的营销活动**
- 需要多实体数据（产品、商店、预订）
- 需要准确的预发送计数
- 具有拆分和扩充的多步批次处理

### 步骤2：验证您的选择

| 您的需要 | 推荐的方法 | 原因 |
|-----------|---------------------|-----|
| 欢迎新客户通过多步入门 | 历程 | 实时入口、多个接触点、条件路径 |
| 每月向订阅者发送新闻稿 | 操作营销活动 | 向受众发送简单的计划消息 |
| 具有提醒序列的购物车放弃 | 历程 | 实时触发器、等待时间、条件跟进 |
| 面向所有客户的促销公告 | 操作营销活动 | 一次性消息，即时投放 |
| 根据行为重新吸引非活动用户 | 历程 | 由受众鉴别、个性化路径触发 |
| 业务事件触发的闪购 | 历程（业务事件） | 影响多个客户的实时触发器 |
| 通过产品目录集成进行季节性促销 | 精心策划的营销活动 | 多实体数据、复杂分段、精确计数 |
| API触发的事务型消息 | API触发的营销活动 | 外部系统触发器，立即投放 |
| 每个预订的多级发送 | 精心策划的营销活动 | 多实体关系，每个预订一条消息 |

## 解释主要区别 {#key-distinctions}

### 历程： 1:1实时编排

**其独特之处：**
- 每个配置文件维护各自的状态和上下文
- 用户档案会按照自己的进度进入和进度
- 基于行为和事件的实时决策
- 等待活动创建个性化计时
- 条件分支为每个配置文件创建唯一路径

**示例流：**

```
Customer A: Abandoned cart → Wait 2 hours → No purchase? → Send reminder → Purchased? → End
Customer B: Abandoned cart → Wait 2 hours → Already purchased → End immediately
```

每个客户都会根据其操作体验自己的历程时间线。

[了解有关历程的更多信息](../building-journeys/journey.md)

### 营销活动：简单的批量投放或触发式投放

**其独特之处：**
- 所有配置文件均以相同方式同时处理
- 无状态执行 — 不维护上下文
- 简单计划或API触发
- 非常适合于广播通信

**示例流：**

```
Monday 9 AM → Send newsletter to 100,000 subscribers → All receive simultaneously
```

每个人都同时收到同一条消息。

**类型：**
- **操作活动**：计划提交（一次性或循环）
- **API触发的营销活动**：通过来自外部系统的API调用触发

[了解有关营销活动的更多信息](../campaigns/get-started-with-campaigns.md)

### 编排的营销活动：批量画布工作流

**其独特之处：**
- 包含活动和过渡的批次画布（类似于历程画布，但面向批次）
- 多实体关系数据支持（配置文件+产品+商店+预订）
- 在画布中构建按需受众
- 发送前的准确计数（发送前可见性）
- 多级发送（每个实体一条消息，例如，每个预订）
- 所有配置文件一起批量处理

**示例流：**

```
Query customers → Filter by purchase history → Split by region → 
Enrich with product data → Build segments → Send personalized offers → All in one batch execution
```

将工作流的复杂性与批量活动执行结合使用。

[详细了解编排的营销活动](../orchestrated/gs-orchestrated-campaigns.md)

## 用例示例 {#use-cases}

### 历程用例

- **购物车放弃恢复**：由购物车添加事件触发，等待结帐，如果没有购买，则发送提醒
- **客户入门**：包含基于配置文件数据的个性化内容的多步欢迎系列
- **忠诚度级别升级**：当客户达到新级别时触发，发送祝贺和福利
- **生日营销活动**：根据生日输入的内容，个性化优惠
- **重新参与**：由受众资格（不活动）、渐进式外展触发

### Campaign用例（操作和API触发）

**操作营销活动：**
- **每月新闻稿**：已计划批量交付给订阅者区段
- **促销公告**：面向目标受众的时效性优惠
- **产品发布**：协调所有客户的发布
- **季节性问候语**：特定日期的假日消息

**API触发的营销活动：**
- **订单确认**：购买后由电子商务系统触发
- **送货通知**：由物流系统触发
- **帐户警报**：欺诈检测系统触发
- **密码重置**：由应用程序中的用户操作触发

### 编排的活动用例

- **具有目录集成的季节性促销活动**：查询产品目录、识别符合条件的客户、按首选项进行分段、发送个性化产品推荐
- **特定于商店的营销活动**：使用商店库存数据定位接近特定商店位置的客户
- **多次预订通信**：每次预订（酒店预订、航班预订）发送一条消息
- **复杂的区段编排**：通过从多个数据源扩充逐步构建受众
- **发送前验证**：在启动主要营销活动之前获取收件人的确切计数

## 常见问题 {#common-questions}

**问：营销策略中是否可以结合历程和营销活动？**

答：当然可以！ 大多数组织针对不同的方案使用所有三种方法：
- 行为、实时参与的历程
- 计划广播通信的操作营销活动
- API触发的事务性消息营销活动
- 为复杂、数据密集型批量活动编排活动

**问：我是否可以将营销活动转换为历程，反之亦然？**

答：不可以，必须以适当的格式重建体验。 但是，您可以重用内容、受众和逻辑概念。

**问：哪种方法更容易构建？**

答：操作营销活动通常最简单（向受众发送单个消息），其次是API触发的营销活动、历程（使用多步骤逻辑更复杂）和编排的营销活动（由于画布工作流和多实体功能而最复杂）。

**问：哪些缩放比例更适合大受众？**

答：这三者都可以很好地扩展，但是：
- **读取受众历程**&#x200B;和&#x200B;**操作营销活动**&#x200B;已针对大型批量受众进行优化
- **协调的营销活动**&#x200B;在包含大型数据集的复杂分段方面表现出色
- **单一历程**&#x200B;单独处理配置文件，因此规模取决于事件数量

**问：我是否可以在历程和营销活动中使用相同的受众？**

答：是，在Adobe Experience Platform中创建的受众可以跨所有三种方法使用。
