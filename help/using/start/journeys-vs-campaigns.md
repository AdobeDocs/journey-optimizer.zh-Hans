# 历程与促销活动：选择正确的方法 {#journeys-vs-campaigns}

>[!BEGINSHADEBOX]

**在此页面上：**&#x200B;将历程与操作和API触发的营销活动进行比较，以便您能够为Adobe Journey Optimizer中的每个营销用例选择正确的方法。 有关协调的营销活动，请参阅[开始使用协调的营销活动](../orchestrated/gs-orchestrated-campaigns.md)。

>[!ENDSHADEBOX]

[!DNL Adobe Journey Optimizer]提供两种主要方法来联系和吸引您的客户：**历程**&#x200B;和&#x200B;**促销活动**。 历程专为受客户行为驱动的实时、多步编排而设计，而营销活动更适合向定义的受众进行一次性或计划式广播。 确定营销活动后，您可以选择最适合您的用例的营销活动类型。

本指南可帮助您根据执行样式、数据需求和用例在历程、操作营销活动和API触发的营销活动之间进行选择 — 并提供快速比较、决策树和具体示例。

>[!NOTE]
>
>**协调的营销活动**&#x200B;具有独特的体系结构特征（中心端批处理执行、多实体关系数据），需要专门的指导。 为了避免过度简化，它们不包括在下面的比较表中。 [了解关于编排的营销活动的更多信息](../orchestrated/gs-orchestrated-campaigns.md)

## 快速比较概述 {#quick-overview}

| 方法 | 最适合 | 执行样式 |
|----------|----------|-----------------|
| **历程** | 带条件逻辑的多步实时客户体验 | 1:1编排 — 每个用户档案按自己的步调 |
| **操作营销活动** | 对受众的计划或定期广播 | 批量执行 — 在发送时一起处理的受众 |
| **API触发的营销活动** | 来自外部系统的事件驱动或事务型消息 | 按需执行 — 由具有负载的API调用触发 |

>[!TIP]
>
>**经验法则：**&#x200B;需要每个客户按照自己的进度使用实时逻辑吗？ 使用&#x200B;**历程**。 是否按计划向受众发送一条消息？ 使用&#x200B;**操作营销活动**。 是否通过API触发来自外部系统的单个消息？ 使用&#x200B;**API触发的营销活动** — 或者&#x200B;**单一事件历程**（如果您在API发送事件后需要多步骤编排）。

## 详细比较 {#detailed-comparison}

使用本综合表了解主要区别：

| 功能 | 历程 | 操作营销活动 | API 触发的营销活动 |
|---------|----------|------------------|------------------------|
| **主要目的** | 使用实时客户上下文进行多步骤1:1编排 | 向受众投放一次性或定期消息 | 由外部系统发起的事务性或事件驱动型消息 |
| **画布类型** | 1:1画布 — 每个配置文件按各自的进度运行 | 无画布 — 单个操作执行 | 无画布 — 单个操作执行 |
| **执行流** | 顺序操作，配置文件维护整个历程的状态 | 同时执行到整个受众 | 每个API调用立即执行 |
| **进入机制** | 事件、受众、资格、业务事件 | 手动激活和计划 | 来自外部系统的API调用 |
| **数据模型** | 实时个人资料+事件数据 | 来自Experience Platform受众的个人资料数据 | 具有可选配置文件查找的API有效负荷数据 |
| **区段** | 预建受众+实时条件 | Experience Platform中的预建受众 | 有效负载驱动的定位（无计划受众） |
| **配置文件正在处理** | 单个实时（当事件发生时） | 批次，一次完成 | 每个API调用，有效负载驱动 |
| **个性化** | 实时上下文数据+配置文件属性 | 轮廓属性 | 有效负载数据+可选配置文件属性 |
| **复杂性** | 具有分支、等待时间和条件的多步 | 单个操作或简单工作流 | 具有有效负荷映射的单个操作 |
| **最适合** | 客户生命周期历程、入门、购物车放弃 | 促销活动、快讯、公告 | 订单确认、送货通知、密码重置 |
| **计时** | 连续，发布后始终有效 | 计划的开始/结束日期 | 通过API提供的按需事件驱动型 |
| **状态管理** | 维护实时操作的客户状态 | 无状态执行 | 每个调用的无状态执行 |
| **使用时间** | 需要具有实时决策逻辑的多个接触点 | 在特定时间向受众发送简单消息 | 外部系统需要立即触发消息 |
| **独特的功能** | 实时反应、等待活动、基于用户档案的步调 | 计划、受众定位、速率控制 | API有效负载映射，系统到系统触发 |

## 决策指南 {#decision-guide}

按照此诊断树选择正确的方法。 许多品牌使用多种类型；请为每个用例选择最适合的类型。

### 步骤1：您的执行要求是什么？

**个人对客户行为的实时响应？**
→ **使用历程**
* 用户档案需要按照自己的速度移动
* 基于行为的条件逻辑
* 实时上下文至关重要

**在计划时间向受众发送简单邮件？**
→ **使用操作营销活动**
* 所有用户档案都会同时接收消息
* 计划的或定期的发送
* 无需复杂的多步逻辑

**外部系统触发的即时消息？**
→ **使用API触发的营销活动**（单消息） **或单一事件历程**（多步骤编排）
* 通过API调用按需触发 — 营销活动发送一条消息；单一历程通过[Experience Platform摄取](../event/additional-steps-to-send-events-to-journey.md)摄取事件并运行完整历程流
* 有效负荷驱动的个性化
* 无需多步逻辑时选择营销活动

**具有高级分段、多实体数据或精确预发送计数的复杂批处理工作流？**
→ **使用协调的营销活动** — 有关详细指导，请参阅[协调的营销活动入门](../orchestrated/gs-orchestrated-campaigns.md)。

### 步骤2：验证您的选择

| 您的需要 | 推荐的方法 | 原因 |
|-----------|---------------------|-----|
| 欢迎新客户通过多步入门 | 历程 | 实时入口、多个接触点、条件路径 |
| 每月向订阅者发送新闻稿 | 操作营销活动 | 向受众发送简单的计划消息 |
| 具有提醒序列的购物车放弃 | 历程 | 实时触发器、等待时间、条件跟进 |
| 面向所有客户的促销公告 | 操作营销活动 | 一次性消息，即时投放 |
| 根据行为重新吸引非活动用户 | 历程 | 由受众鉴别、个性化路径触发 |
| 业务事件触发的闪购 | 历程（业务事件） | 影响多个客户的实时触发器 |
| API触发的事务型消息（单次发送） | API 触发的营销活动 | 外部系统触发器，即时一次性投放 |
| API触发的多步流程 | 历程（单一事件） | 外部系统通过API发送单一事件；Journey协调后续步骤 |
| 具有多实体数据的复杂批处理工作流 | 编排的营销活动 | 请参阅[开始使用编排的营销活动](../orchestrated/gs-orchestrated-campaigns.md) |

## 解释主要区别 {#key-distinctions}

### 历程： 1:1实时编排

**其独特之处：**
* 每个配置文件维护各自的状态和上下文
* 用户档案会按照自己的进度进入和进度
* 基于行为和事件的实时决策
* 等待活动创建个性化计时
* 条件分支为每个配置文件创建唯一路径
* 内置的活动侦听 — 在定义的时间段内不操作也会触发下一步，而不仅仅是显式事件。 [了解等待活动](../building-journeys/wait-activity.md)
* 频率封顶 — 控制客户可在历程中输入或接收消息的频率。 [了解历程上限](../conflict-prioritization/journey-capping.md)
* 按百分比拆分受众 — 将用户档案分为基于百分比的随机组，以便跨历程路径运行A/B实验。 [了解百分比拆分](../building-journeys/condition-activity.md)
* 测试模式 — 在发布实时之前，使用测试用户档案验证历程逻辑和消息投放。 [了解测试模式](../building-journeys/testing-the-journey.md)

**示例流：**

```
Customer A: Abandoned cart → Wait 2 hours → No purchase? → Send reminder → Purchased? → End
Customer B: Abandoned cart → Wait 2 hours → Already purchased → End immediately
```

每个客户都会根据其操作体验自己的历程时间线。

[了解有关历程的更多信息](../building-journeys/journey.md)

### 营销活动：简单的批量投放或触发式投放

**其独特之处：**
* 所有配置文件均以相同方式同时处理
* 无状态执行 — 不维护上下文
* 简单计划或API触发
* 非常适合于广播通信

**示例流：**

```
Monday 9 AM → Send newsletter to 100,000 subscribers → All receive simultaneously
```

每个人都同时收到同一条消息。

**类型：**
* **操作营销活动**：计划向受众投放（一次性或循环）
* **API触发的营销活动**：由包含有效负载数据的API调用触发的按需交付

[了解有关营销活动的更多信息](../campaigns/get-started-with-campaigns.md)

## 用例示例 {#use-cases}

### 历程用例

* **购物车放弃恢复**：由购物车添加事件触发，等待结帐，如果没有购买，则发送提醒
* **客户入门**：包含基于配置文件数据的个性化内容的多步欢迎系列
* **忠诚度级别升级**：当客户达到新级别时触发，发送祝贺和福利
* **生日营销活动**：根据生日输入的内容，个性化优惠
* **重新参与**：由受众资格（不活动）、渐进式外展触发

### Campaign用例（操作和API触发）

**操作营销活动：**
* **每月新闻稿**：已计划批量交付给订阅者区段
* **促销公告**：面向目标受众的时效性优惠
* **产品发布**：协调所有客户的发布
* **季节性问候语**：特定日期的假日消息

**API触发的营销活动：**
* **订单确认**：购买后由电子商务系统触发
* **送货通知**：由物流系统触发
* **帐户警报**：欺诈检测系统触发
* **密码重置**：由应用程序中的用户操作触发

## 常见问题 {#common-questions}

+++ 营销策略中是否可以结合使用历程和营销活动？

可以。 许多组织会针对不同的方案使用所有方法：

* 用于行为实时参与的&#x200B;**历程**
* 计划广播通信的&#x200B;**操作营销活动**
* 针对事务性消息&#x200B;**API触发的营销活动**
* **为复杂、数据密集型批处理营销活动编排的营销活动** — 请参阅[开始使用编排的营销活动](../orchestrated/gs-orchestrated-campaigns.md)

针对每个用例使用正确的工具，而不是为所有用例强制使用一种方法。

+++

+++ 我是否可以将营销活动转换为历程，反之亦然？

否，必须以适当的格式重建体验。 但是，您可以重用内容、受众和逻辑概念。

+++

+++ 哪种方法更容易构建？

操作营销活动通常最简单（向受众发送单个消息），其次是API触发的营销活动，然后是历程（使用多步骤逻辑会更加复杂）。

+++

+++ 哪些缩放比例更适合大受众？

这三种模式都可以很好的扩展；正确的选择取决于您的模式：

* **读取受众历程**&#x200B;和&#x200B;**操作营销活动**&#x200B;已针对大型批量受众（一条消息或同时流向多个用户档案）进行了优化。
* **单一（基于事件）历程**&#x200B;在事件发生时单独处理配置文件，因此缩放取决于事件数量和吞吐量。

有关具有大型数据集和多实体数据的复杂分段，请参阅[编排的营销活动](../orchestrated/gs-orchestrated-campaigns.md)。

+++

+++ 我是否可以在历程和营销活动中使用相同的受众？

可以。 在[!DNL Adobe Experience Platform]中创建的受众可用于历程、操作营销活动和编排的营销活动。 API触发的营销活动是有效负载驱动的，不使用相同方式的预建受众。

+++
